# PamojaData Authentication and Authorization Architecture Design

## 1. Overview
This document outlines the architecture for a production-grade authentication and authorization system for the PamojaData humanitarian intelligence platform. The design satisfies the requirements for enterprise-level security, scalability, multi-role access control, and maintainability.

## 2. Authentication System Architecture

### 2.1 Password Security
- **Algorithm:** PBKDF2 with SHA-256
- **Iterations:** 320,000 (industry standard minimum)
- **Salt:** 32 bytes of cryptographically secure random data per user
- **Storage Format:** `algorithm$iterations$salt$hash`

### 2.2 Session Management
- **Token Type:** Cryptographically secure random tokens (32 bytes)
- **Refresh Strategy:** Dual-token system (Session Token + Refresh Token)
- **Expiration:** Session tokens expire after 8 hours; Refresh tokens expire after 7 days
- **Binding:** Sessions are bound to User Agent and IP Address for hijack detection
- **Revocation:** Centralized session table allows immediate invalidation across all devices

### 2.3 Protection Mechanisms
- **Brute-Force Protection:** Account lockout after 5 failed attempts (30-minute duration)
- **Spam Prevention:** Honeypot fields on registration forms
- **Verification:** Mandatory email verification before login is permitted

## 3. Role-Based Access Control (RBAC)

### 3.1 Role Hierarchy
The system implements a flexible, database-driven RBAC model with the following default roles:
1. **Admin:** Full system access, user management, global settings
2. **Staff/Manager:** Operational access across all programmes
3. **Moderator:** Quality assurance and read-only analytics
4. **Programme Manager:** Programme-specific management
5. **M&E Officer:** Data collection and quality assessment
6. **Standard User:** Core workflows and basic data input
7. **Donor:** Read-only dashboard access

### 3.2 Permission Granularity
Permissions are granular and mapped to roles via a junction table (`role_permissions`). This allows creating new roles without changing application code.

### 3.3 Enforcement Layers
1. **UI Level:** Streamlit sidebar dynamically hides inaccessible pages
2. **Page Level:** Each page handler verifies permissions before rendering
3. **Action Level:** Critical actions (e.g., user deletion) verify permissions explicitly

## 4. Secure Admin Creation Workflow

To prevent privilege escalation vulnerabilities, the system implements an **Admin Approval Workflow** combined with **Secret Invite Keys**.

### 4.1 The Approach
Public signup is restricted to the "Standard User" role. Elevated roles require an invite token generated by an existing Admin.

### 4.2 Why this is the best architecture:
- **Security:** Zero chance of an attacker manipulating a POST request to become an admin
- **Auditability:** Every elevated account is explicitly tied to the admin who invited them
- **Usability:** Streamlines onboarding for legitimate staff without manual database intervention

## 5. Database Schema

### 5.1 Users Table
```sql
CREATE TABLE users (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    username TEXT UNIQUE NOT NULL,
    full_name TEXT NOT NULL,
    email TEXT UNIQUE NOT NULL,
    phone TEXT,
    org_name TEXT,
    password_hash TEXT NOT NULL,
    is_active INTEGER DEFAULT 1,
    is_suspended INTEGER DEFAULT 0,
    is_banned INTEGER DEFAULT 0,
    email_verified INTEGER DEFAULT 0,
    failed_login_attempts INTEGER DEFAULT 0,
    locked_until TIMESTAMP,
    last_login TIMESTAMP,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 5.2 RBAC Tables
```sql
CREATE TABLE roles (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT UNIQUE NOT NULL,
    description TEXT
);

CREATE TABLE permissions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT UNIQUE NOT NULL,
    description TEXT
);

CREATE TABLE role_permissions (
    role_id INTEGER,
    permission_id INTEGER,
    PRIMARY KEY (role_id, permission_id),
    FOREIGN KEY (role_id) REFERENCES roles(id) ON DELETE CASCADE,
    FOREIGN KEY (permission_id) REFERENCES permissions(id) ON DELETE CASCADE
);

CREATE TABLE user_roles (
    user_id INTEGER,
    role_id INTEGER,
    assigned_by INTEGER,
    assigned_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (user_id, role_id),
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    FOREIGN KEY (role_id) REFERENCES roles(id) ON DELETE CASCADE,
    FOREIGN KEY (assigned_by) REFERENCES users(id) ON DELETE SET NULL
);
```

### 5.3 Session & Audit Tables
```sql
CREATE TABLE sessions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NOT NULL,
    session_token TEXT UNIQUE NOT NULL,
    refresh_token TEXT UNIQUE NOT NULL,
    expires_at TIMESTAMP NOT NULL,
    refresh_expires_at TIMESTAMP NOT NULL,
    ip_address TEXT,
    user_agent TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);

CREATE TABLE audit_logs (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER,
    action TEXT NOT NULL,
    details TEXT,
    ip_address TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE SET NULL
);
```

## 6. Security Hardening

- **SQL Injection:** All database queries use parameterized statements (`?` placeholders)
- **XSS:** Streamlit automatically escapes HTML output; `unsafe_allow_html` is restricted to static CSS/UI elements
- **CSRF:** Streamlit handles session state securely, mitigating CSRF
- **Broken Authentication:** Strong hashing, account lockout, and secure session generation
- **Sensitive Data:** Passwords are never logged or exposed in API responses
